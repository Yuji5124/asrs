# ASRS AI連携方針（Phase 5）

## 基本思想

**AIは「JSONを読み書きする外部の共作者」として扱う。**
ツール内部にAIを組み込むのではなく、境界面（JSONスキーマとvalidate）をきれいに保つことで、後からClaude Code / Codexなどと連携しやすくする。

AIは全自動生成のためではなく、**ユーザーが作りたいRPGを作るための制作補助**に使う。

## AIに支援させたいこと

- NPC会話案を作る
- イベントコマンド案を作る
- 敵データを作る
- アイテムやスキルのバランス案を出す
- 小さいマップ案を作る
- クエスト案を作る
- JSONデータの不整合をチェックする
- ゲームとして破綻していないかレビューする

## 連携の仕組み（設計方針）

1. **スキーマを契約にする。** `data-schema.md` と `src/core/types/` が常に一致している状態を保つ。AIへの依頼は「このスキーマに従うJSONを出力せよ」の形になる。
2. **validateを独立した純関数にする。** 手入力・ファイル読み込み・AI出力の**すべてが同じvalidateを通る**。AI専用の検査経路を作らない。
3. **部分インポートを可能にする。** 「敵3体だけ」「イベント1個だけ」をID単位でマージできる構造にする（IDが人間可読slugである理由）。
4. **差分プレビュー → 承認 → 反映。** AI出力は直接反映しない。必ず「取り込み前に差分を見せて、ユーザーが承認してから反映」のフローを挟む。これがAI連携の安全弁。
5. **プロジェクトのコンテキスト化。** 「世界観・トーン・既存の敵/アイテム一覧」を1つのMarkdown/JSONに要約してAIに渡せるエクスポート機能を用意する。
6. **プロンプトテンプレート集。** よく使う依頼（NPC会話案・敵データ案・バランス調整案など）を `prompts/` にテンプレートとして蓄積する。

## 段階的な導入

- **Phase 1〜4:** 手動連携で十分。「プロジェクトJSONをコピーしてClaude Codeに貼り、返ってきたJSONをインポートする」。API組み込みを急がない。
- **Phase 5:** 取り込みUI（validate → 差分プレビュー → 承認）と、validate強化（ID参照切れ・到達不能マップ・進行破綻のチェック）を実装する。
- **それ以降（構想）:** エディタ内からのAI呼び出し。ただし外部API依存はオプションに留め、APIなしでもツールが完結することを維持する。

## AI出力の受け入れ基準

- スキーマvalidateを通ること（通らない出力は自動で差し戻し）
- 既存IDと衝突しないこと（衝突時はリネーム提案）
- `originality-policy.md` に違反する固有名詞を含まないこと（レビュー観点に含める）
